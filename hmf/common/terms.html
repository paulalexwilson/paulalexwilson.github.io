compact IRI
A compact IRI has the form of prefix:suffix and is used as a way of expressing an IRI without needing to define separate term definitions for each IRI contained within a common vocabulary identified by prefix.
datatype IRI
A datatype IRI is an IRI identifying a datatype that determines how the lexical form maps to a literal value.
IRI
The absolute form of an IRI containing a scheme along with a path and optional query and fragment segments.
IRI reference
Denotes the common usage of an Internationalized Resource Identifier.
annotation
An annotation holds supplimentary metadata about properties and child nodes that is not typically part of the core domain model. See [[[#annotations]]].
base version
The base version is the version from which a given node was checked-out. Changes between the working tree node and its base version constitute the set of versionable changes.
child node
A child node C of a given node P is one where there exists a parent-child relationship between P and C where P is the parent and C is the child.
child nodes type definitions
A child node type definition is used to permit and constrain a child on a node within a node type definition. See [[[#child-node-definition]]].
default namespace
The default namespace is the convenient immutable empty namespace that requires no term to be specified in the local name. See [[[#empty-namespace]]].
document
A document is a tree serialized using one of the HMF vocabularies.
domain model
The domain is the real-world subject that is being formalized with HMF.
effective type
The effective type of a node is the set of all item definitions inherited from its supertypes and mixin types.
empty namespace
See default namespace.
entity
An entity is anything held within the repository that has an identity. This can be regular nodes within the working tree but may also refer to objects such as the namespace registry and repository node. See [[[#entities]]].
formalism
A formalism is the process of encoding the rules and characterstics of a domain using the HMF type system.
frozen node
A frozen node is a node within version storage that tracks the versioned state of a versionable node. See [[[#frozen-node]]].
identity
An identity is a means to unambigously distinguish an object from another through the use of an identifier. HMF uses an Internationalized Resource Identifier to represent identity within the model.
local name
A local name distinguishes a name within its namespace and ideally provides a reasonable and understandable label to the item.
location
Location refers to the position of node within the tree. A location is a way to resolve a node independently of its identity by traversing from the root node.
items
An item is the part of a tree that can be located using a path. Informally it is either a node or a property.
item definition
An item definition is either a property definition or child node definition.
mixin
A mixin type is a type that can be composed onto a node (or onto a primary type) to provide additional metadata or functionality. See [[[#mixin-types]]].
name
A name uniquely identifies an item within a tree and consists of a namespace and a local name.
namespace
A namespace is an entity to which local names belong allowing them to be uniquely and unambigously identified. They allow data arriving from independent sources to coexist witin the same tree without risk naming collisions. They also allow meaning and semantics to be attached to items within the tree by virtue of their membership within the namespace.
namespace IRI
A namespace IRI is an IRI that identifies a namespace. A namespace IRI may be aliased using a term to provide a short hand name. See [[[#names]]].
namespace registry
The namespace registry provides the mapping between terms and namespace IRIs. See [[[#namespace-registry]]]
navigation
Navigation is the process to following relationships between nodes. Traversal is a form of navigation following parent-child relationships.
node
Nodes provide the structure to a metadata tree modelled within HMF. They are named and typed entities acting as containers of properties. See [[[#nodes]]].
node identity
A node identity provides a what to unique identify a node within a HMF tree using its IRI. See [[[#node-identity]]].
node type definitions
Defines a node type within the model in terms of property type definitions and node type definitions. See [[[#node-type-definition]]].
on-parent-version
The on-parent-version is a value applied to a node type that defines how a node is treated when its parent versionable is checked-in. See [[[#on-parent-version]]].
parent-child relationship
A parent-child relationship is a type of relationship that forms the metadata tree that indicate a containment or ownership relationship between nodes. See [[[#parent-child-relationships]]].
path
A path is a sequence of segments allowing an item to be resolved within a tree via navigation from one node to another. In the case of an absolute path the node is resolved from the root node. In the case of a relative path the node is resolved from an arbitrary node. See [[[#paths]]] and [[[#path-segments]]].
prefix
Prefix is a synonym of term.
primary item
The primary item of a node is an item which can be retrieved without knowing its name. See [[[#primary-item]]].
primary type
The primary type of a node defines its type and therefore core item definitions. See [[[#primary-type]]].
property
Properties hold values on nodes and provide the information about the domain modelled by a given HMF tree.
property type definitions
A property type definition is used to permit and constrain a property on a node within a node type definition. See [[[#property-definition]]].
reference
A reference is a type of unidirectional relationship between nodes. They can either by strong or weak as explained in [[[#references]]].
referenceable
A node is referenceable if it can be the target of a reference. Only nodes that are referenceable may be the subject of a reference. See [[[#referenceable-type]]].
relationship
A relationship connects two nodes. They can be bidirectional as in the case of parent-child relationships or unidirectional as in the case of reference relationships.
repository
A repository provides storage and operation semantics to metadata trees within HMF. See [[[#repository]]].
repository node
The repository node provides metadata about the repository. See [[[#repository-node]]].
residual item definition
A residual item definition is an item definition that covers all remaining items defined on a node type. See [[[#residual-property-definition]]] and [[[#residual-child-node-definition]]].
resource
A resource is an instance of a given entity, typically transformed into a representation for transfer via an API.
root
The root is the base or the metadata tree and acts as the ultimate ancestor of all nodes. Absolute paths are anchored from the root.
same name siblings
A same name sibling is a pair of child nodes with the same name. HMF does not support same name siblings.
segment
A segment is a component of a path used to locate nodes. See [[[#path-segments]]].

supertype
A supertype is a node type from which a node type definition inherits its item definitions.
term
A term is the short-hand alias of a namespace used in compact names.
traversal
Traversal is the process of navigating from the root node and a node using a path. See [[[#traversal]]].
tree
A tree is a set of nodes connected with parent-child relationships. A connected subgraph of a tree is referred to as a subtree.
type
A type is classification for a node that defines the set of permitted item definitions it may contain.
value
A value holds the data of a property and is subject to a value type and value constraints. They can be either single-valued or multi-valued.
value constraint
A value constraint restricts the range of possible values for a value.
value type
A value type defines the data type of values held by a property identified by a datatype IRI. Common examples are type:string and type:long. See [[[#property-required-type]]].
versionable
A versionable is a type of node that may be stored within version storage as a version. Only versionable nodes may be checked-in or checked-out. See [[[#versioning]]] and specifically [[[#versionable-nodes]]].
version
A version is a node held in version storage that represents a checked-in version of a node held in the working tree. See [[[#version-metadata]]].
version label
A version label is a string identifier used to locate a version. See [[[#version-labels]]].
version history
The version history of a versionable node is the set of all its versions. See [[[#version-history]]].
version storage
A region of metadata outside of the working tree for the storage of tracked versions of nodes.
versionable change
A versionable change is a change made to a versionable in the working tree that represents a change to the base version.
versioned reference
A versioned reference is a reference that uses an annotation to specify an additional version label allowing clients to resolve that reference to a specific version.
versionable state
The versionable state of a versionable includes the versionable node and a subset of its descendent subtree depending on the on-parent-version state of its child nodes.
vocabulary
A vocabulary is a concrete syntax for the representation of HMF trees on disk or on the wire.
working tree
The working tree represents the current head state of the metadata against which changes are written and read. The working tree is contrasted with other entities such as those present within version storage or repository-specific entities such as the namespace registry or the repository node. See [[[#working-tree]]].
